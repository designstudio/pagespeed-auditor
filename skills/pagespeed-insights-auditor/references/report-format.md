# Report Format

Use this file when reading or explaining the artifacts generated by the PageSpeed skill.

## Engines

This skill now has two audit engines.

### `local-lighthouse`

Primary engine. Use it first when the goal is to run a real PageSpeed-style audit from inside the repository.

What it gives you:

- Lighthouse scores for `performance`, `accessibility`, `best-practices`, and `seo`
- Real metrics like FCP, LCP, TBT, CLS, Speed Index, and INP
- Opportunities and diagnostics from the local run
- JSON and HTML artifacts for each strategy
- Repeatable before-vs-after comparisons on localhost or preview servers

### `psi-remote`

Optional follow-up engine. Use it only when you want Google-hosted PSI validation for a public URL.

What it adds:

- the public Google PSI perspective
- optional CrUX-aware follow-up when available
- external validation of the deployed route

## Files

A local run contains:

- `index.md`: consolidated report across all audited URLs.
- `<target-slug>/mobile.lighthouse.json`: raw Lighthouse JSON for the mobile strategy.
- `<target-slug>/mobile.lighthouse.html`: raw Lighthouse HTML for the mobile strategy.
- `<target-slug>/desktop.lighthouse.json`: raw Lighthouse JSON for the desktop strategy.
- `<target-slug>/desktop.lighthouse.html`: raw Lighthouse HTML for the desktop strategy.
- `<target-slug>/summary.md`: human-readable report for a single audited URL.

A PSI follow-up run can still contain:

- `index.md`
- `<target-slug>/mobile.json`
- `<target-slug>/desktop.json`
- `<target-slug>/summary.md`

## Report Modes

Treat each report as belonging to one of these modes:

- `baseline`: first clean local Lighthouse run for a route set.
- `multi-route`: same run, several important routes.
- `compare`: rerun after fixes with deltas versus a previous baseline.
- `budget-check`: same audit data, but budget pass/fail is emphasized.
- `psi-followup`: optional public PSI pass after the local run.
- `blocked-run`: PSI follow-up did not complete, so the report records the blocker while keeping the local audit as primary evidence.

Mention the engine and mode near the top of the report.

## Preferred Output Shape

For successful local Lighthouse runs, structure the report like this:

```markdown
**Run Status**
Engine: local-lighthouse
Mode: baseline / multi-route / compare / budget-check
Execution: completed
Target: ...

**Route Summary**
| Strategy | Performance | Accessibility | Best Practices | SEO |

**Metrics**
| Strategy | FCP | LCP | TBT | CLS | SI | INP |

**Top Issues**
1. ...
2. ...
3. ...

**Code Mapping Leads**
- ...
- ...

**Raw Artifacts**
- ...

**Next Pass**
- ...
```

For PSI follow-up or blocked PSI runs, keep the same general shape but make the engine and execution status explicit.

## Confirmed Findings vs Repo-Backed Hypotheses

Keep these separate.

- `Engine-confirmed findings`: scores, metrics, opportunities, diagnostics, and measured deltas returned directly by Lighthouse or PSI.
- `Repo-backed hypotheses`: likely causes inferred from the codebase, build output, route architecture, asset strategy, framework behavior, and bundle composition.

Do not present hypotheses as if the audit engine had directly confirmed them.

## What Local Lighthouse Reports Should Contain

### Run status

Always include:

- which engine ran,
- which mode the report represents,
- whether the run completed,
- whether comparison mode was active,
- which route set was audited.

### Scores

Each strategy should list the Lighthouse category scores returned by the engine:

- `performance`
- `accessibility`
- `best-practices`
- `seo`

Treat these as Lighthouse scores on a 0-100 scale.

### Core metrics

Per-route reports should include at least:

- FCP
- LCP
- TBT
- CLS
- Speed Index
- INP

Use these to explain the shape of the performance problem, not just the final score.

### Opportunities and diagnostics

The report should separate:

- highest-impact opportunities,
- selected diagnostics,
- optional budget failures,
- optional score deltas versus a previous run.

Estimated milliseconds and bytes are directional, not exact promises.

### Code mapping leads

Each report should include a short section that converts the strongest Lighthouse findings into investigation leads inside the repo.

Examples:

- `unused-javascript`: inspect large client bundles, broad imports, icon packs, and route-level chunking.
- `render-blocking-resources`: inspect font loading, CSS entrypoints, preload usage, and synchronous scripts.
- `modern-image-formats` or `uses-responsive-images`: inspect asset formats, image components, and responsive source usage.
- `server-response-time`: inspect hosting, route handlers, SSR, and document latency.
- `dom-size` or `mainthread-work-breakdown`: inspect large page trees, hydration scope, and expensive client render paths.

## PSI Follow-Up Guidance

Use PSI follow-up only when you want the public Google-hosted perspective after a successful local run.

Good use cases:

- validating a production URL
- checking whether the deployed route behaves differently from localhost
- gathering public-route evidence after local fixes

## Blocked-Run Guidance

Use blocked-run mode when PSI follow-up could not complete because of issues such as:

- `429 RESOURCE_EXHAUSTED`
- anonymous quota exhaustion
- missing or invalid API key
- missing public URL
- network failure reaching PSI

When blocked-run mode is active:

- say clearly that local Lighthouse succeeded or failed separately from PSI,
- record the PSI blocking reason,
- keep the local Lighthouse findings as the primary audit evidence,
- continue with repo-based performance investigation only when useful,
- include explicit rerun instructions.

A blocked PSI follow-up should still be useful, but it must not look like the whole audit failed if the local Lighthouse pass succeeded.

## Comparisons

When `--compare-dir` is used, show score deltas against the previous run for matching route slugs and strategies.

Use comparisons to verify whether a change actually improved the route instead of relying on intuition.

## Budgets

When budgets are provided, mark each configured category as:

- `pass`
- `fail`
- `missing`

Budget failures are a prioritization tool, not proof of a specific root cause.

## How to Explain Results

- Lead with the biggest user-facing issue, not the longest audit list.
- Mention whether the problem is mostly mobile, mostly desktop, or both.
- If accessibility or SEO scores are low, mention the weakest audit areas instead of only quoting the score.
- When multiple routes are audited, identify whether the issue is route-specific or systemic.
- In blocked-run mode, lead with the blocker and then the most actionable fallback findings.

## Mapping Findings Back to Code

Use the strongest engine findings as leads, then inspect the repo for concrete causes.

When reporting suggestions, separate:

- engine-confirmed findings,
- repo-backed root-cause hypotheses,
- code changes that are safe to implement now,
- changes that need product or infra decisions.
