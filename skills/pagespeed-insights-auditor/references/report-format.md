# Report Format

Use this file when reading or explaining the artifacts generated by `scripts/run_pagespeed_insights.py`.

## Files

Each run contains:

- `index.md`: consolidated report across all audited URLs.
- `<target-slug>/mobile.json`: raw PSI response for the mobile strategy.
- `<target-slug>/desktop.json`: raw PSI response for the desktop strategy.
- `<target-slug>/summary.md`: human-readable report for a single audited URL.

## Report Modes

Treat each report as belonging to one of these modes:

- `baseline`: first clean PSI run for a route set.
- `multi-route`: same run, several important routes.
- `compare`: rerun after fixes with deltas versus a previous baseline.
- `budget-check`: same PSI data, but budget pass/fail is emphasized.
- `blocked-run`: PSI did not complete, so the report switches to a repo-based fallback structure.

Mention the mode near the top of the report when it helps the reader understand what kind of evidence they are looking at.

## Preferred Output Shape

For successful PSI runs, structure the report like this:

```markdown
**Run Status**
PSI: Completed
Mode: baseline / multi-route / compare / budget-check
Target set: ...

**Route Summary**
| Route | Strategy | Performance | Accessibility | Best Practices | SEO |

**Top Issues**
1. ...
2. ...
3. ...

**Repo-Backed Hypotheses**
- ...
- ...

**Recommended Fixes**
1. ...

- ...
```

For blocked PSI runs, structure the report like this:

```markdown
**Run Status**

**Static Codebase Findings**
1. ...
2. ...
3. ...

- ...
```

## PSI-Confirmed Findings vs Repo-Backed Hypotheses

Keep these separate.

- `PSI-confirmed findings`: scores, metrics, field data, opportunities, diagnostics, and measured deltas directly returned by PSI.
- `Repo-backed hypotheses`: likely causes inferred from the codebase, build output, route architecture, asset strategy, framework behavior, and bundle composition.

Do not present hypotheses as if PSI had confirmed them.

## What Successful PSI Reports Should Contain

### Run status

Always include:

- whether PSI completed,
- which mode the report represents,
- whether the run used an API key or anonymous quota,
- whether comparison mode was active,
- which route set was audited.

### Scores

Each strategy lists the Lighthouse category scores returned by PSI:

- `performance`
- `accessibility`
- `best-practices`
- `seo`

Treat these as Lighthouse scores on a 0-100 scale.

### Core metrics

Per-route reports should include at least:

- FCP
- LCP
- CLS
- INP
- TBT
- Speed Index

Use these to explain the shape of the performance problem, not just the final score.

### Field data

When Chrome UX Report data is present, the report should include:

- Largest Contentful Paint
- Interaction to Next Paint
- Cumulative Layout Shift
- overall origin or URL assessment

If field data is missing, say the route is being judged from lab data only.

### Opportunities and diagnostics

The report should separate:

- highest-impact opportunities,
- selected diagnostics,
- optional budget failures,
- optional score deltas versus a previous run.

Estimated milliseconds and bytes are directional, not exact promises.

## Blocked-Run Guidance

Use blocked-run mode when PSI could not complete because of issues such as:

- `429 RESOURCE_EXHAUSTED`
- anonymous quota exhaustion
- missing or invalid API key
- missing public URL
- network failure reaching PSI

When blocked-run mode is active:

- say clearly that PSI did not complete,
- record the blocking reason,
- continue with repo-based performance investigation,
- prioritize only high-confidence codebase findings,
- include explicit rerun instructions.

A blocked run should still be useful, but it must not look like a finished PSI audit.

## Comparisons

When `--compare-dir` is used, show score deltas against the previous run for matching route slugs and strategies.

Use comparisons to verify whether a change actually improved the route instead of relying on intuition.

## Budgets

When budgets are provided, mark each configured category as:

- `pass`
- `fail`
- `missing`

Budget failures are a prioritization tool, not proof of a specific root cause.

## How to Explain Results

- Lead with the biggest user-facing issue, not the longest audit list.
- Mention whether the problem is mostly mobile, mostly desktop, or both.
- If accessibility or SEO scores are low, mention the weakest audit areas instead of only quoting the score.
- When multiple routes are audited, identify whether the issue is route-specific or systemic.
- In blocked-run mode, lead with the blocker and then the most actionable fallback findings.

## Mapping PSI Findings Back to Code

Use the PSI findings as leads, then inspect the repo for concrete causes.

Examples:

- Poor LCP: inspect hero sections, route-level data loading, server rendering delays, font loading, above-the-fold images.
- Unused JavaScript: inspect large client bundles, analytics SDKs, client-only wrappers, icon packs, wide barrel imports.
- Render-blocking resources: inspect font declarations, synchronous script tags, stylesheet order, global CSS, third-party embeds.
- Image findings: inspect image component usage, asset dimensions, responsive sources, lazy loading, next-gen formats.
- Cache findings: inspect asset hosting strategy, framework headers, CDN config, immutable asset naming.

When reporting suggestions, separate:

- PSI-confirmed findings,
- repo-backed root-cause hypotheses,
- code changes that are safe to implement now,
- changes that need product or infra decisions.
