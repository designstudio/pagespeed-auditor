# Report Format

Use this file when reading or explaining the artifacts generated by `scripts/run_pagespeed_insights.py`.

## Files

Each run now contains:

- `index.md`: consolidated report across all audited URLs.
- `<target-slug>/mobile.json`: raw PSI response for the mobile strategy.
- `<target-slug>/desktop.json`: raw PSI response for the desktop strategy.
- `<target-slug>/summary.md`: human-readable report for a single audited URL.

## What the reports contain

### Scores

Each strategy lists the Lighthouse category scores returned by PSI:

- `performance`
- `accessibility`
- `best-practices`
- `seo`

Treat these as Lighthouse scores on a 0-100 scale.

### API key status

The reports should record whether the run used:

- an API key loaded from the configured environment variable, or
- anonymous PSI quota.

Mention anonymous quota when reporting results because repeated automation may become rate-limited.

### Core metrics

Per-route reports include at least:

- FCP
- LCP
- CLS
- INP
- TBT
- Speed Index

Use these to explain the practical shape of the performance problem, not just the final score.

### Field data

When Chrome UX Report data is present, the report includes:

- Largest Contentful Paint
- Interaction to Next Paint
- Cumulative Layout Shift
- overall origin or URL assessment

Use field data to explain what real users experience. Distinguish it from lab-only Lighthouse results.

### Opportunities and diagnostics

The report separates:

- highest-impact opportunities,
- selected diagnostics,
- optional budget failures,
- optional score deltas versus a previous run.

Estimated milliseconds and bytes are directional, not exact promises.

## Comparisons

When `--compare-dir` is used, the report should show score deltas against the previous run for matching route slugs and strategies.

Use comparisons to verify whether a change actually improved the route instead of relying on intuition.

## Budgets

When budgets are provided, the report should mark each configured category as:

- `pass`
- `fail`
- `missing`

Budget failures are a prioritization tool, not proof of a specific root cause.

## How to explain results

- Lead with the biggest user-facing issue, not the longest audit list.
- Mention whether the problem is mostly mobile, mostly desktop, or both.
- If field data is missing, say the report is based on lab data only.
- If accessibility or SEO scores are low, mention the weakest audit areas instead of only quoting the score.
- When multiple routes are audited, identify whether the issue is route-specific or systemic.

## Mapping PSI findings back to code

Use the PSI findings as leads, then inspect the repo for concrete causes.

Examples:

- Poor LCP: inspect hero sections, route-level data loading, server rendering delays, font loading, above-the-fold images.
- Unused JavaScript: inspect large client bundles, analytics SDKs, client-only wrappers, icon packs, wide barrel imports.
- Render-blocking resources: inspect font declarations, synchronous script tags, stylesheet order, global CSS, third-party embeds.
- Image findings: inspect image component usage, asset dimensions, responsive sources, lazy loading, next-gen formats.
- Cache findings: inspect asset hosting strategy, framework headers, CDN config, immutable asset naming.

When reporting suggestions, separate:

- findings supported directly by PSI,
- repo-backed root-cause hypotheses,
- code changes that are safe to implement now,
- changes that need product or infra decisions.
